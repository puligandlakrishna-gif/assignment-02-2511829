ETL Decisions
Decision 1 — Normalising Mixed Date Formats
Problem: The date column in the source CSV stored dates in three different formats across rows — 29/08/2023 (DD/MM/YYYY), 12-12-2023 (DD-MM-YYYY), and 2023-02-05 (ISO 8601). Casting this column directly to a DATE type would fail or silently misparse values, and sorting or comparing dates across formats produces wrong results.
Resolution: In the ETL transform stage, each value was passed through a format-detection routine that tried each pattern in sequence and parsed whichever matched. All dates were then written to the warehouse in ISO 8601 (YYYY-MM-DD) exclusively. The dim_date surrogate key (date_key) was derived from this cleaned value as an integer in YYYYMMDD form, ensuring that chronological sort order is preserved even without a JOIN to the date dimension.
Decision 2 — Standardising Inconsistent Category Casing
Problem: The category field contained logically identical values written differently across rows: electronics, Electronics, Grocery, and Groceries all appear in the source data. Grouping by category in any BI query would treat these as four distinct buckets, splitting revenue that belongs together and making category-level reports incorrect.
Resolution: Every category value was uppercased and trimmed first, then mapped through a small canonical lookup: GROCERIES → Grocery, ELECTRONICS → Electronics, CLOTHING → Clothing. The cleaned values were stored in dim_product.category, and a CHECK constraint on the column enforces that only the three canonical strings can ever be loaded — so the same problem cannot re-enter the warehouse from a future load.
Decision 3 — Backfilling NULL Store Cities
Problem: 19 rows (6.3% of the dataset) had a blank store_city field. Leaving these NULL would cause those transactions to be excluded from any city-level or regional analysis, silently understating revenue for the affected stores.
Resolution: The store name was always populated and each of the five store names maps unambiguously to exactly one city (Chennai Anna → Chennai, Mumbai Central → Mumbai, and so on). A static lookup table keyed on store_name was used during the transform stage to backfill the missing city values before loading. The corrected city was then stored in dim_store.city, which is declared NOT NULL, making it impossible for future loads to insert a store row without a city.
