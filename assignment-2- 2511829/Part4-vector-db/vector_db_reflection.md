Would Keyword Search Suffice?
No — and the mismatch is fundamental, not incidental.
Traditional keyword search works by matching the exact terms in a query against an index of terms in the documents. A lawyer asking "What are the termination clauses?" would get results only where the word "termination" literally appears. But contracts are drafted in varied legal language. The same concept might appear as "right to exit," "dissolution of agreement," "either party may discontinue," or "notice of cancellation." Keyword search misses all of these. Worse, it has no way to understand that the question is asking for a specific type of clause — it treats the query as a bag of words, not an intent.
There are three specific failures that make keyword search inadequate here:
Vocabulary mismatch. Legal contracts use formal, archaic, and jurisdiction-specific language that rarely matches how a lawyer naturally phrases a question. A query for "what happens if a party goes bankrupt" will not surface a clause that reads "upon the occurrence of an Insolvency Event as defined in Schedule 4."
No semantic understanding. Keyword systems rank results by term frequency and co-occurrence (TF-IDF, BM25). They have no model of meaning. A clause about "termination for convenience" and a clause about "automatic renewal" both discuss how a contract ends, but a keyword search on "termination" would surface the first and miss the second entirely.
500-page scale with no structure. A contract of this length contains hundreds of distinct clauses, cross-references, definitions, and schedules. Keyword search returns matching lines, not semantically coherent clause units. The lawyer still has to manually read surrounding context to understand what was found — which defeats the purpose of the system.
The Role of a Vector Database
A vector database solves this by shifting from term matching to meaning matching. The pipeline works in three stages:
1. Chunking and embedding at index time. The 500-page contract is split into meaningful chunks — individual clauses, paragraphs, or defined sections. Each chunk is passed through a language model (such as all-MiniLM-L6-v2 or a legal-domain model like LegalBERT) which converts it into a dense vector — a point in high-dimensional space where proximity encodes semantic similarity. These vectors are stored in a vector database (Pinecone, Weaviate, pgvector, etc.).
2. Query embedding at search time. When a lawyer types "What are the termination clauses?", that question is embedded by the same model into a vector in the same space. The question and the relevant contract clauses now occupy nearby regions of that space — regardless of whether they share any words.
3. Approximate nearest-neighbour retrieval. The vector database runs an ANN search (FAISS, HNSW, etc.) to find the chunks whose vectors are closest to the query vector by cosine similarity. The top results — the semantically most relevant clauses — are returned in milliseconds even across a 500-page document.
This means a query about termination will surface not just clauses containing the word "termination," but also clauses about exit rights, notice periods, automatic expiry, and insolvency triggers — because the model has learned that these concepts cluster together in meaning.
Why This Matters Specifically for Legal Work
The retrieval layer is typically paired with a generative model (a RAG architecture — Retrieval-Augmented Generation). The retrieved clauses are passed as context to an LLM, which then composes a direct answer: "Section 14.2 allows either party to terminate with 30 days' written notice. Section 14.5 provides for immediate termination upon a material breach that is not remedied within 10 business days." The lawyer gets a precise, grounded answer with source references — not a list of matching lines to sift through.
The vector database is the component that makes this trustworthy: it ensures the model is answering from the actual contract text rather than from its training data, and it localises retrieval to the specific clauses most relevant to the question asked.
In short, keyword search tells you where a word appears. A vector database tells you where an idea appears — and for a lawyer navigating dense contractual language, that distinction is the entire value of the system.
